# 유스케이스 문서

## 1. 액터 정의
### 주요 액터
- 비회원 방문자: 회원가입, 로그인, 이벤트 탐색을 수행한다.
- 회원: 프로필 관리, 대기열 이용, 좌석 선택, 예매 생성, 결제, 구매/환불 조회, 결제 포기를 수행한다.
- 매니저: 자신이 생성한 이벤트의 등록/수정/삭제/운영, 구매 조회, 환불 처리를 수행한다.
- 관리자: 백오피스에 로그인해 사용자 조회, 새 사용자 생성, 매니저 승급, 논리 삭제, 정지, 정지 해제를 수행한다.

### 보조 액터
- 소셜 로그인 제공자: OAuth 인증을 완료한다.
- 결제 제공자: Toss Payments 결제 승인/취소를 수행한다.
- 파일 업로드 클라이언트: 이미지 업로드 URL로 이벤트 이미지를 전송한다.

## 3. 유스케이스 상세
## UC-01. 비회원 방문자가 회원가입한다
**액터**
- 비회원 방문자

**보조 액터**
- 없음

**목표**
- 예매 기능을 사용할 수 있는 USER 계정을 생성한다.

**사전조건**
- 비회원 방문자는 로그인하지 않은 상태다.

**기본 흐름**
1. 비회원 방문자가 이메일, 비밀번호, 이름, 나이, 성별, 전화번호, 지역 정보를 입력한다.
2. 시스템은 입력 형식을 검증한다.
3. 시스템은 유효한 가입 요청으로 계정을 생성한다.
4. 시스템은 생성된 계정에 기본 역할 `USER`를 부여한다.
5. 시스템은 가입 요청 결과와 생성 대상 사용자 식별자를 반환한다.

**예외 흐름**
- 입력 형식이 올바르지 않으면 시스템은 오류를 반환한다.
- 이미 존재하는 사용자 정보이면 시스템은 가입을 거절한다.

**결과**
- 비회원 방문자는 USER 역할의 로그인 가능한 계정을 생성한다.

**비기능 요구사항**
- 개인정보 처리 및 입력 검증이 필요하다.

## UC-02. 비회원 방문자가 로그인한다
**액터**
- 비회원 방문자

**보조 액터**
- 소셜 로그인 제공자

**목표**
- 인증된 사용자 상태가 되어 보호된 기능을 사용할 수 있다.

**사전조건**
- 비회원 방문자는 유효한 계정을 가지고 있거나 소셜 로그인 제공자 인증이 가능해야 한다.

**기본 흐름**
1. 비회원 방문자가 이메일/비밀번호 로그인 또는 소셜 로그인을 선택한다.
2. 이메일 로그인인 경우 시스템은 자격 증명을 검증한다.
3. 소셜 로그인인 경우 비회원 방문자는 제공자 인증을 완료한다.
4. 시스템은 소셜 인증 이메일을 기준으로 기존 계정을 찾는다.
5. 같은 이메일의 계정이 있으면 시스템은 소셜 인증 정보를 기존 계정과 연동한다.
6. 같은 이메일의 계정이 없으면 시스템은 USER 역할의 신규 계정을 생성한다.
7. 시스템은 로그인 성공 결과와 인증 유지에 필요한 정보를 제공한다.

**예외 흐름**
- 이메일 또는 비밀번호가 올바르지 않으면 로그인에 실패한다.
- 소셜 로그인 인증 결과가 유효하지 않으면 로그인에 실패한다.
- 토큰 재발급에 필요한 정보가 무효하면 재발급에 실패한다.

**결과**
- 사용자는 인증된 세션 상태를 획득하거나, 실패 사유를 전달받는다.

**비기능 요구사항**
- 인증/토큰 저장 보안 정책이 필요하다.

## UC-03. 비회원 방문자가 이벤트를 탐색한다
**액터**
- 비회원 방문자

**보조 액터**
- 없음

**목표**
- 예매 가능한 이벤트를 찾고 상세 정보를 확인한다.

**사전조건**
- 없음

**기본 흐름**
1. 비회원 방문자가 이벤트 목록 조회를 요청한다.
2. 필요하면 키워드, 카테고리, 지역, 가격 범위, 상태, 날짜 범위를 지정한다.
3. 시스템은 `OPEN` 상태이며 조회 조건에 맞는 이벤트 목록을 반환한다.
4. 비회원 방문자가 특정 이벤트 상세를 조회한다.
5. 시스템은 생성일, 예매 가능 기간, 공연 시작일, 좌석/가격/상태 정보를 포함한 이벤트 상세를 반환한다.
6. 필요하면 시스템은 카테고리 목록과 이벤트 이미지를 함께 제공한다.

**예외 흐름**
- 잘못된 필터 파라미터가 전달되면 시스템은 오류를 반환한다.
- 존재하지 않거나 `DELETED` 상태인 이벤트를 조회하면 시스템은 실패를 반환하거나 노출하지 않는다.
- 이미지 파일이 존재하지 않으면 시스템은 파일을 반환하지 않는다.

**결과**
- 비회원 방문자는 관심 이벤트 탐색 결과와 상세 정보를 확인한다.

**비기능 요구사항**
- 이벤트 상세 조회 시 조회수 증가가 응답을 과도하게 지연시키지 않아야 한다.
- 주요 API 응답은 기대 기준 1초 미만을 목표로 한다.

## UC-06. 회원이 예매 대기열에 진입한다
**액터**
- 회원

**보조 액터**
- 없음

**목표**
- 혼잡한 이벤트 예매를 위해 예매 순서를 획득하고, 슬롯이 비면 예매 입장 토큰을 받는다.

**사전조건**
- 회원이 로그인되어 있어야 한다.
- 대상 이벤트가 `OPEN` 상태이고 예매 가능 기간 안에 있어야 한다.
- 공연 시작일이 아직 도래하지 않아야 한다.

**기본 흐름**
1. 회원이 특정 이벤트의 polling 대기열 진입을 요청한다.
2. 시스템은 회원을 대기열에 등록한다.
3. 회원은 polling API로 현재 대기 상태 또는 순번을 조회한다.
4. 예매 가능 슬롯이 비면 시스템은 다음 회원을 승급한다.
5. 시스템은 승급된 회원에게 입장 토큰을 발급한다.
6. 회원은 입장 토큰으로 좌석 선택과 예매 생성을 진행할 수 있다.

**예외 흐름**
- 인증되지 않은 사용자는 대기열에 진입할 수 없다.
- 이벤트가 예매 가능 기간 밖이거나 공연 시작일이 도래했으면 대기열 진입을 거절한다.
- 재접속은 허용하지 않으며, 사용자는 대기열 처음부터 다시 진입해야 한다.
- polling 요청이 과도하면 시스템은 rate control 정책에 따라 제한할 수 있다.

**결과**
- 회원은 대기열 상태를 확인하고, 승급 시 예매 진행에 필요한 입장 토큰을 획득한다.

**비기능 요구사항**
- 동시 대기열 사용자 5,000명을 목표로 한다.
- polling은 SWR, single-flight, server-side polling rate control을 전제로 한다.

## UC-07. 회원이 예매 대기열에서 이탈한다
**액터**
- 회원

**보조 액터**
- 없음

**목표**
- 더 이상 예매를 진행하지 않을 때 대기 상태를 종료한다.

**사전조건**
- 회원이 로그인되어 있어야 한다.
- 회원이 대기열에 등록되어 있어야 한다.

**기본 흐름**
1. 회원이 대기열 이탈을 요청한다.
2. 시스템은 회원의 대기열 등록 상태를 확인한다.
3. 시스템은 대기열 등록 상태를 종료한다.
4. 슬롯이 비어 다음 사용자를 승급할 수 있으면 시스템은 다음 회원을 승급 대상으로 만든다.
5. 시스템은 대기열 이탈 결과를 반환한다.

**예외 흐름**
- 인증되지 않은 사용자는 이탈 요청을 수행할 수 없다.
- 이미 대기열에서 이탈한 회원이면 시스템은 현재 상태를 반환한다.

**결과**
- 회원은 대기열에서 이탈하고 더 이상 해당 대기 순번을 사용하지 않는다.

**비기능 요구사항**
- 대기열 이탈 처리는 다음 사용자 승급과 충돌하지 않아야 한다.

## UC-08. 회원이 좌석을 선택해 예매를 생성한다
**액터**
- 회원

**보조 액터**
- 없음

**목표**
- 특정 이벤트의 좌석을 선택해 1시간 안에 결제해야 하는 결제 대기 예매를 생성한다.

**사전조건**
- 회원이 로그인되어 있어야 한다.
- 회원은 유효한 입장 토큰을 보유해야 한다.
- 대상 이벤트가 예매 가능 기간 안에 있어야 한다.

**기본 흐름**
1. 회원이 특정 이벤트의 좌석 배치 조회를 요청한다.
2. 시스템은 현재 좌석 배치를 반환한다.
3. 회원이 좌석 ID 목록과 티켓 수량을 제출해 예매 생성을 요청한다.
4. 시스템은 회원의 입장 토큰, 이벤트 예매 가능 상태, 좌석 가능 상태를 확인한다.
5. 시스템은 회원이 이미 보유한 결제 대기 예매가 없는지 확인한다.
6. 시스템은 선택한 좌석과 수량을 기준으로 결제 대기 예매를 생성한다.
7. 시스템은 결제 제한시간 1시간을 부여한다.
8. 시스템은 생성된 예매 식별자와 결제 가능 종료 시각을 반환한다.

**예외 흐름**
- 인증 정보가 없거나 역할이 맞지 않으면 요청을 거절한다.
- 입장 토큰이 없거나 만료되었으면 예매 생성을 거절한다.
- 예매 생성 커밋 시점에 이벤트 상태 변경이 감지되면 시스템은 이벤트 변경을 고지하고 예매 생성을 롤백한다.
- 회원이 이미 결제 대기 예매를 1개 보유하고 있으면 새로운 예매 생성을 거절한다.

**결과**
- 회원은 결제 전 단계의 예매를 생성하거나, 생성 실패 사유를 전달받는다.

**비기능 요구사항**
- 좌석 중복 판매 방지 동시성 제어가 필요하다.
- 회원당 동시에 보유 가능한 결제 대기 예매는 전체 1개로 제한된다.

- 결제 준비 금액과 선택 좌석 총액이 불일치하면 결제를 거절한다.

- 잘못된 결제 정보이면 승인 실패 또는 미완료 상태가 반환될 수 있다.

**결과**
- 회원은 생성된 예매에 대한 결제 접수 결과를 확인하고, 성공 시 활성 예매를 획득한다.

## 4. 시스템 전역 비기능 요구사항
### 4.1 성능
- 이벤트 조회는 Projection 기반으로 제공하고 상세 조회의 조회수 증가는 비동기로 처리한다.
- 대기열은 동시 사용자 5,000명을 목표로 한다.
- 모든 API 응답시간은 p95 기준 1초 미만을 유지한다.
- 모든 API는 통일된 처리 목표로 API별 100 TPS를 기준으로 설계한다.
- 사용자에게 노출되는 오류 메시지는 내부 사유를 과도하게 드러내지 않고 `사용자 오류` 또는 `서버 오류` 수준으로 축약한다.

### 4.2 확장성
- Gateway, Eureka, Redis, RabbitMQ, 서비스 분리를 전제로 서비스 단위 확장이 가능해야 한다.
- polling 대기열은 SWR, single-flight, server-side polling rate control을 전제로 한다.
- 브로커, 결제, 이벤트, 좌석 서비스는 동일 API 처리 목표인 100 TPS와 p95 1초 미만 응답 목표를 기준으로 병목을 관찰하고 필요 시 수평 확장할 수 있어야 한다.

### 4.3 가용성
- 특정 모듈 장애가 전체 예매 기능 중단으로 즉시 확산되지 않도록 서비스 경계를 유지한다.
- SSE 대기열은 deprecated이며 공식 가용성 기준은 polling 대기열이다.
- 대기열/예매/결제 장애 시 사용자에게는 `서버 오류` 수준의 축약 메시지만 노출한다.

### 4.4 데이터 정합성
- 예매 생성 또는 결제 승인 커밋 시점에 이벤트 상태 변경이 감지되면 시스템은 이벤트 변경을 고지하고 해당 트랜잭션을 롤백해야 한다.
- 결제 승인 도중 예매가 만료되면 시스템은 결제를 취소해야 한다.
- 결제 제한시간 초과 시 예매 자동 취소와 좌석 즉시 해제가 반영되어야 한다.

### 4.5 동시성 제어
- 시스템은 유효한 입장 토큰이 없는 예매 생성/예매 결제를 차단해야 한다.
- 시스템은 좌석 중복 판매 방지 규칙이 동시 요청에서도 유지되도록 해야 한다.
- 대기열 승급, 토큰 만료, 결제 완료, 예매 만료가 동시에 발생해도 상태가 중복 확정되지 않아야 한다.

### 4.6 보안
- 시스템은 인증 및 역할 기반 접근 제어를 적용해야 한다.
- 게이트웨이는 공개 엔드포인트와 보호 엔드포인트를 구분해야 한다.
- 매니저는 자신이 생성한 이벤트에 대해서만 운영 권한을 가져야 한다.
- 관리자 백오피스 사용자 관리 기능은 관리자 역할을 가진 사용자에게만 허용해야 한다.
- 관리자 로그인 실패 횟수 제한은 3회다.
- 개인정보 암호화는 비밀번호에만 적용한다.
- 관리자 로그인 실패가 3회 발생하면 관리자 계정을 잠근다.
- 관리자 백오피스 인증/인가는 세션 방식으로 구현한다.
- 관리자 세션은 30분 동안 비활성 상태이면 만료된다.

### 4.7 장애 대응 및 복구
- 대기열 이탈, 예매 만료, 결제 상태 조회를 통해 사용자가 장애 또는 지연 상태를 인지할 수 있어야 한다.
- 결제 제한시간 초과 시 예매는 자동 취소되고 좌석은 즉시 해제되어야 한다.
- 결제/환불 처리가 서버 비정상 종료 등으로 실패하면 DLQ를 사용해 실패 작업을 재처리해야 한다.
- 이미지 업로드 또는 교체 저장 실패 시 3회 재시도 후 대체 이미지를 노출하고 사용자에게 축약 오류를 알려준다.
- 이미지 실패 사용자 알림은 추후 구현한다.

### 4.8 감사 추적 및 운영성
- 시스템은 Swagger/OpenAPI, Actuator, Prometheus를 통해 운영 상태를 점검할 수 있어야 한다.
- 대기열 사용자 수, polling 요청률, 활성 입장 토큰 수, 결제 처리 상태, 예매 만료 수, 환불 처리 결과를 관찰할 수 있어야 한다.
- 관리자 백오피스의 사용자 생성, 매니저 승급, 논리 삭제, 정지, 정지 해제 이력은 감사 추적 가능해야 한다.
- 관리자 백오피스 UI는 Thymeleaf로 구현한다.
- 인증/인가가 필요한 API 호출은 요청 원문, 응답 원문, 에러 원문, 사용자 정보, 발생 시각을 감사 로그로 저장한다.
- 감사 로그와 운영 로그는 1개월 동안 보존한다.
- 기존 운영 로그가 있는 경우 유지하고, 부족한 운영 로그는 추가한다.
