# 요구사항

## 1. Scope

- ChangeSet delivery scope: 기존 persisted notification 기능을 운영 가능한 MVP 범위로 확정한다.
- Delivery outcome: 인증된 알림 수신자는 gateway 경유 REST API로 자신의 persisted notification을 조회하고 삭제할 수 있으며, 관리자/매니저는 특정 사용자에게 수동 알림을 생성할 수 있고, 기존 결제/환불 outcome listener는 중복 없이 persisted notification을 계속 적재한다.
- Primary actor(s): 알림 수신자, 알림 발신자(ADMIN/MANAGER)
- Included use cases (1-4): UC-030 알림 수신자가 자신의 알림함을 조회한다, UC-031 알림 수신자가 자신의 알림을 삭제한다, UC-032 관리자 또는 매니저가 특정 사용자에게 알림을 생성한다.
- Supporting / prerequisite work items: 기존 RabbitMQ payment/refund outcome listener 경로 유지, 동일 source payload replay 시 중복 notification row 방지, `/api/v1/notifications/**` 인증/권한/소유권 검증 유지.
- Hard out-of-scope boundary: SSE/실시간 push 전달, public test notification endpoint, 신규 notification 채널 추가, Notification BC 밖 User/Profile 존재 조회 의존성 추가, 신규 publisher 또는 별도 구독 UX 추가.

## 2. Functional Requirements

### FR-001

- Requirement: 시스템은 인증된 알림 수신자에게 자신의 persisted notification 전체 목록, 읽지 않음 목록, 읽지 않음 개수, 단건 상세를 gateway 경유 REST API로 제공해야 한다.
- Success condition: 목록은 최신순 페이지네이션으로 반환되고, 읽지 않음 조회와 개수 조회는 읽음 상태를 바꾸지 않으며, 단건 상세 성공 시 해당 알림 1건만 읽음 상태로 전환된다.
- Failure policy: 미인증 요청은 거절하고, 존재하지 않거나 타인 소유인 notification 상세 요청은 실패시키며 읽음 상태를 변경하지 않는다.

### FR-002

- Requirement: 시스템은 인증된 알림 수신자에게 자신의 persisted notification 삭제를 단건, 선택 집합, 전체 범위로 제공해야 한다.
- Success condition: 요청 범위가 요청자 소유 규칙을 만족하면 해당 owned notification만 삭제하고 삭제 결과를 반환한다.
- Failure policy: 미인증 요청은 거절하고, 타인 소유 notification이 포함된 요청은 거절하며, 선택 삭제에서 이미 없는 owned notification은 그대로 두고 존재하는 owned notification만 삭제한다.

### FR-003

- Requirement: 시스템은 인증된 ADMIN 또는 MANAGER가 특정 수신자 userId, notification type, title, content, optional targetUrl로 알림 1건을 생성할 수 있게 해야 한다.
- Success condition: 유효한 입력이면 persisted notification 1건이 unread 상태로 저장되고 이후 해당 수신자 inbox에서 조회 가능하다.
- Failure policy: 미인증 또는 USER 권한 요청은 거절하고, blank target userId 또는 필수 입력 누락/오류는 저장 없이 실패한다.

### FR-004

- Requirement: 시스템은 기존 payment/refund outcome listener 입력으로 생성되는 persisted notification 경로를 계속 운영 가능하게 해야 한다.
- Success condition: 승인된 기존 listener payload는 persisted notification을 저장하고, 동일 payload replay는 deterministic source key 기준으로 중복 row를 만들지 않는다.
- Failure policy: malformed payload는 notification을 만들지 않고 실패를 격리하며, 실시간 push 성공 여부와 무관하게 persisted inbox 적재 성공 여부를 기준으로 본다.

## 3. Non-Functional Requirements

### NFR-001

- Candidate requirement: 조회/삭제는 authenticated recipient ownership isolation을 보장하고, 생성은 ADMIN/MANAGER role만 허용해야 한다.
- Confirmation status: 확정

### NFR-002

- Candidate requirement: 이번 범위의 delivery surface는 persisted inbox REST API와 persisted create path로 제한하며 SSE, public test endpoint, 신규 채널은 포함하지 않는다.
- Confirmation status: 확정

### NFR-003

- Candidate requirement: 알림 관련 API는 기존 전역 API 정책을 따라 p95 1초 미만, API별 100 TPS 목표로 관찰 가능해야 한다.
- Confirmation status: 기존 전역 정책 적용으로 확정

### NFR-004

- Candidate requirement: 인증/인가 보호 알림 API의 감사 로그와 운영 로그는 기존 전역 정책에 따라 1개월 보존해야 한다.
- Confirmation status: 기존 전역 정책 적용으로 확정
