# 요구사항 명세서

## 1. 개요
- 초기 아이디어: 현재 저장소의 백엔드 API 표면을 기준으로 티켓 예매 플랫폼의 요구사항을 외부 액터 관점에서 정리한다.
- 문제 상황: 티켓 예매 도메인의 가입, 이벤트 탐색, 대기열, 좌석 예매, 결제, 환불, 이벤트 운영 정책이 코드와 구두 정책으로 분산되어 있어 이벤트 스토밍 전 기준 문서가 필요하다.
- 목표: 사용자/매니저/관리자/외부 제공자 관점의 기능 요구사항, 비기능 요구사항, 확정된 비즈니스 정책, 남은 기술 결정을 분리해 다음 설계 단계의 입력으로 사용한다.
- 범위:
  - 인증/회원가입/로그인/토큰 재발급/로그아웃/소셜 로그인
  - 관리자 백오피스 로그인, 사용자 조회/생성/승급/삭제/정지
  - 사용자 프로필 조회/수정
  - 이벤트 목록/상세/카테고리/이미지 조회
  - 매니저 이벤트 등록/수정/삭제/상태 변경/이미지 업로드 및 교체
  - polling 기반 대기열 진입, 순번 조회, 이탈, 입장 토큰 발급
  - 좌석 조회, 결제 대기 예매 생성, 결제 완료 예매 확정
  - 예매 기반 결제 준비/승인/상태 조회/사용자 결제 포기
  - 구매 내역/환불 내역/매니저 환불 처리
  - 범위 제외: 내부 서비스 간 전용 API, 메시지 컨슈머, 배치 전용 테스트 컨트롤러, 프론트엔드 구현, 3D/렌더링, 클라이언트 E2E 테스트, 알림 기능 구현
- 제외/지연:
  - 알림 조회/생성/삭제/실시간 구독은 현재 구현 범위에서 제외한다. 추후 요구사항으로 재개할 수 있지만 이번 이벤트 스토밍 입력에는 포함하지 않는다.
  - SSE 대기열 사용자 경험은 deprecated로 간주하고, 공식 사용자 경험은 HTTP polling 기반 대기열로 정의한다.

## 2. 액터 및 이해관계자
- 주요 액터:
  - 비회원 방문자: 회원가입, 로그인, 이벤트 탐색을 수행한다.
  - 회원: 프로필 관리, 대기열 진입, 좌석 선택, 예매 생성, 결제, 구매/환불 조회, 결제 포기를 수행한다.
  - 매니저: 자신이 생성한 이벤트를 등록/수정/삭제/운영하고, 자신의 이벤트 구매 내역 조회와 환불 처리를 수행한다.
  - 관리자: 백오피스에 로그인해 사용자 계정을 조회하고, 새 사용자 생성, 매니저 승급, 논리 삭제, 정지, 정지 해제를 수행한다.
- 보조 액터:
  - 소셜 로그인 제공자: Google, Kakao OAuth 인증을 제공한다.
  - 결제 제공자: Toss Payments 결제 승인/취소에 필요한 외부 결제 처리를 담당한다.
  - 파일 업로드 클라이언트: 이벤트 이미지 업로드 URL에 바이너리 파일을 업로드한다.
- 이해관계자:
  - 플랫폼 운영팀: 계정 권한, 이벤트 판매 상태, 환불 운영 정책을 관리한다.
  - 고객지원/정산 담당자: 구매 및 환불 이력 조회 결과에 의존한다.
  - 인프라/운영팀: 게이트웨이, 대기열, Redis, RabbitMQ, MySQL, 정적 파일 제공 운영을 담당한다.

## 3. 기능 요구사항
### 3.1 인증 및 계정
- FR-001. 비회원 방문자는 이메일, 비밀번호, 이름, 나이, 성별, 전화번호, 지역 정보를 제출해 회원가입을 요청할 수 있어야 한다.
- FR-002. 시스템은 일반 회원가입 성공 시 기본 역할을 항상 `USER`로 부여해야 한다.
- FR-003. 초기 관리자 계정은 시스템에 이미 존재해야 하며, 이후 운영 계정 관리는 관리자가 백오피스에서 수행해야 한다.
- FR-004. 비회원 방문자는 이메일과 비밀번호로 로그인할 수 있어야 한다.
- FR-005. 시스템은 로그인 성공 시 액세스 토큰을 응답 헤더로, 리프레시 토큰을 쿠키로 제공해야 한다.
- FR-006. 로그인한 사용자는 로그아웃할 수 있어야 하며, 시스템은 사용 중이던 리프레시 토큰을 무효화해야 한다.
- FR-007. 비회원 방문자는 소셜 로그인 제공자별 로그인 URL을 조회할 수 있어야 한다.
- FR-008. 시스템은 소셜 최초 가입을 이메일 기준으로 식별해야 하며, 같은 이메일의 기존 계정이 있으면 해당 계정과 연동해야 한다.
- FR-009. 비회원 방문자는 소셜 로그인 콜백을 통해 인증을 완료하고 액세스 토큰/리프레시 토큰을 발급받을 수 있어야 한다.
- FR-010. 사용자는 유효한 토큰 정보가 있을 때 액세스 토큰과 리프레시 토큰을 재발급받을 수 있어야 한다.

### 3.2 사용자 프로필
- FR-011. 인증된 회원 또는 매니저는 자신의 프로필 정보를 조회할 수 있어야 한다.
- FR-012. 인증된 회원은 자신의 이름, 성별, 전화번호, 지역, 나이 등 프로필 정보를 수정할 수 있어야 한다.

### 3.3 이벤트 탐색 및 조회
- FR-013. 비회원 또는 회원은 이벤트 목록을 조회할 수 있어야 한다.
- FR-014. 시스템은 이벤트 목록 조회 시 키워드 검색, 단일/복수 카테고리, 지역, 가격 범위, 이벤트 상태, 날짜 범위 필터를 지원해야 한다.
- FR-015. 시스템은 `OPEN` 상태 이벤트를 사용자에게 노출해야 하며, `DELETED` 상태 이벤트는 사용자에게 노출하지 않아야 한다.
- FR-016. 비회원 또는 회원은 이벤트 상세 정보를 조회할 수 있어야 한다.
- FR-017. 시스템은 이벤트 상세 조회 시 이벤트 식별자, 제목, 썸네일, 생성일, 예매 가능 기간, 공연 시작일, 가격 범위, 상태, 지역, 좌석 수, 잔여 좌석 수 등 판매 판단에 필요한 정보를 제공해야 한다.
- FR-018. 시스템은 이벤트 상세 조회가 발생하면 조회수를 비동기로 증가시켜야 한다.
- FR-019. 비회원 또는 회원은 이벤트 카테고리 목록을 조회할 수 있어야 한다.
- FR-020. 비회원 또는 회원은 이벤트 썸네일/상세 이미지를 정적 URL로 조회할 수 있어야 한다.

### 3.4 이벤트 운영
- FR-021. 인증된 매니저는 제목, 카테고리, 설명, 관람 제한, 썸네일 URL, 좌석 레이아웃, 예매 가능 기간, 공연 시작일, 연령 제한, 좌석 선택 가능 여부를 포함해 이벤트를 등록할 수 있어야 한다.
- FR-022. 시스템은 이벤트 등록/수정 시 예매 가능 기간이 공연 시작일보다 앞서도록 검증해야 하며, 그렇지 않으면 요청을 거절해야 한다.
- FR-023. 시스템은 이벤트가 예매 가능 기간 안에 있고 `OPEN` 상태일 때만 예매 가능하게 처리해야 한다.
- FR-024. 시스템은 공연 시작일이 도래한 이벤트를 자연스럽게 예매 불가 상태로 간주해야 한다.
- FR-025. 인증된 매니저는 자신이 생성한 이벤트 목록을 조회할 수 있어야 한다.
- FR-026. 이벤트를 생성한 매니저만 해당 이벤트 정보를 수정할 수 있어야 한다.
- FR-027. 이벤트를 생성한 매니저만 해당 이벤트 상태를 변경할 수 있어야 한다.
- FR-028. 이벤트를 생성한 매니저만 해당 이벤트를 삭제할 수 있어야 한다.
- FR-029. 인증된 매니저는 업로드할 원본 파일명 목록을 제출해 이벤트 이미지 업로드 URL을 발급받을 수 있어야 한다.
- FR-030. 시스템은 발급한 URL에 대해 `.webp` 확장자, 최대 10MB 제한을 만족하는 이미지만 업로드받아야 한다.
- FR-031. 시스템은 업로드된 이벤트 이미지를 정적 경로에서 다시 조회 가능하게 저장해야 한다.
- FR-032. 시스템은 이벤트 이미지 교체 시 기존 이미지의 버전을 관리하지 않고, 기존 이미지를 삭제한 뒤 새 이미지로 교체해야 한다.

### 3.5 대기열 및 입장 제어
- FR-033. 인증된 회원은 특정 이벤트의 티켓 예매 대기열에 진입할 수 있어야 한다.
- FR-034. 시스템은 공식 대기열 사용자 경험으로 polling 기반 대기열 진입, 현재 순번 조회, 대기열 이탈을 지원해야 한다.
- FR-035. 시스템은 예매 가능 슬롯이 비어 있을 때 대기열의 다음 회원을 승급해야 한다.
- FR-036. 시스템은 회원이 예매 가능 슬롯에 진입하는 시점에 입장 토큰을 발급해야 한다.
- FR-037. 시스템은 입장 토큰이 결제 제한시간 경과 또는 결제 완료 시 만료되도록 처리해야 한다.
- FR-038. 시스템은 대기열 재접속을 허용하지 않아야 하며, 재시도가 필요한 사용자는 대기열 처음부터 다시 진입해야 한다.
- FR-039. 시스템은 기존 SWR, single-flight, server-side polling rate control을 전제로 polling 요청 부하를 제어해야 한다.
- FR-040. 인증된 회원은 더 이상 예매를 진행하지 않을 경우 대기열 등록을 명시적으로 종료할 수 있어야 한다.
- FR-041. 시스템은 유효한 입장 토큰이 있는 사용자에게만 예매 생성과 예매 결제를 허용해야 한다.

### 3.6 좌석 조회 및 예매 생성
- FR-042. 인증된 회원은 특정 이벤트의 좌석 배치를 조회할 수 있어야 한다.
- FR-043. 인증된 회원은 유효한 입장 토큰이 있을 때만 예매 생성을 위한 좌석 선택을 요청할 수 있어야 한다.
- FR-044. 시스템은 회원이 제출한 좌석 ID 목록과 티켓 수량을 기준으로 결제 대기 예매를 생성해야 한다.
- FR-045. 시스템은 결제 대기 예매마다 1시간의 결제 제한시간을 부여해야 한다.
- FR-046. 시스템은 결제 대기 예매가 제한시간 안에 결제 완료되면 예약 완료 및 결제 완료 상태의 활성 예매로 확정해야 한다.
- FR-047. 시스템은 결제 제한시간 내에 결제되지 않은 예매를 만료 예매로 처리해야 한다.
- FR-048. 시스템은 결제 제한시간이 경과하면 예매를 자동 취소하고 선택 좌석을 즉시 해제해야 한다.
- FR-049. 만료된 예매의 사용자는 같은 좌석 또는 이벤트에 재시도할 수 있으나, 반드시 대기열 처음부터 다시 진입해야 한다.

### 3.7 예매 결제 및 구매
- FR-050. 인증된 회원은 자신이 생성한 결제 대기 예매가 존재하고 결제 제한시간이 유효할 때만 결제 준비를 시작할 수 있어야 한다.
- FR-051. 시스템은 생성된 예매 식별자를 기준으로 결제 대상과 금액을 식별하고 구매 준비 결과를 생성해야 한다.
- FR-052. 시스템은 결제 준비 금액과 선택 좌석 총액이 불일치하면 결제 준비 또는 승인을 예외 처리해야 한다.
- FR-053. 인증된 회원은 생성된 예매를 기반으로 만들어진 구매 건에 대해 구매 ID, 결제키, 주문 ID, 결제 금액, 결제 제공자 정보를 제출해 결제 승인을 요청할 수 있어야 한다.
- FR-054. 시스템은 결제 승인 요청을 비동기 처리 대상으로 접수하고, 사용자에게 접수 결과를 반환해야 한다.
- FR-055. 인증된 회원은 결제 상태 polling API로 특정 예매 또는 구매 ID의 결제 승인 처리 상태를 조회할 수 있어야 한다.
- FR-056. 인증된 회원은 자신의 구매 내역을 결제 상태별로 조회할 수 있어야 한다.
- FR-057. 인증된 매니저는 자신이 생성한 이벤트의 구매 내역만 조회할 수 있어야 한다.
- FR-058. 회원의 결제 취소는 결제 전 예매 포기 또는 결제 흐름 중단을 의미해야 하며, 결제 완료 후 금전 반환은 환불로 처리해야 한다.

### 3.8 환불
- FR-059. 사용자는 자신의 환불 내역을 페이지 단위로 조회할 수 있어야 한다.
- FR-060. 사용자는 특정 환불 건의 상세를 조회할 수 있어야 한다.
- FR-061. 관리 목적 사용자는 환불 상태별 목록을 조회할 수 있어야 한다.
- FR-062. 매니저는 자신이 생성한 이벤트의 특정 구매 건에 대해서만 환불을 요청할 수 있어야 한다.
- FR-063. 매니저 환불은 날씨 등 운영 사유로 특정 구매를 취소하는 업무로 정의해야 한다.
- FR-064. 시스템은 부분 환불을 허용하지 않아야 한다.
- FR-065. 매니저는 이벤트 취소 등 상황에서 자신이 생성한 특정 이벤트의 전체 구매 건에 대한 일괄 환불을 요청할 수 있어야 한다.
- FR-066. 시스템은 환불 상태를 `REQUESTED`, `PROCESSING`, `COMPLETED`, `FAILED` 수준으로 구분해 조회할 수 있어야 한다.

### 3.9 관리자 백오피스 사용자 관리
- FR-067. 관리자는 관리자용 백오피스에 로그인할 수 있어야 한다.
- FR-068. 시스템은 관리자 백오피스 접근 시 관리자 역할을 검증해야 한다.
- FR-069. 관리자는 백오피스에서 사용자 목록과 사용자 상세 정보를 조회할 수 있어야 한다.
- FR-070. 관리자는 백오피스에서 새 사용자 계정을 생성할 수 있어야 하며, 운영 계정으로 `MANAGER`와 `ADMIN` 역할 계정을 생성할 수 있어야 한다.
- FR-071. 관리자는 백오피스에서 기존 사용자를 매니저로 승급할 수 있어야 한다.
- FR-072. 관리자는 백오피스에서 사용자를 논리 삭제할 수 있어야 한다.
- FR-073. 관리자는 백오피스에서 사용자를 정지할 수 있어야 한다.
- FR-074. 관리자는 백오피스에서 정지된 사용자의 정지를 해제할 수 있어야 한다.
- FR-075. 시스템은 정지된 사용자의 로그인을 거절해야 한다.
- FR-076. 시스템은 백오피스 사용자 조회 시 이메일은 로컬 파트의 앞 2자와 뒤 2자 및 도메인만 노출하고, 이름은 앞 2자만 노출해야 한다.
- FR-077. 시스템은 관리자 로그인 실패가 3회 발생하면 실패 횟수 초과 상태를 기록하고 관리자 계정을 잠가야 한다.
- FR-078. 시스템은 논리 삭제 또는 정지 상태 사용자의 로그인, 예매, 결제, 환불 조회를 허용하지 않아야 한다.

## 4. 비기능 요구사항
> 사용자가 확정하지 않은 항목은 `후보` 또는 `확인 필요`로 표시한다.

### 4.1 성능
- NFR-001. 이벤트 목록/상세 조회는 Projection 기반 조회와 비동기 조회수 증가 방식을 유지해 읽기 응답을 무겁게 만들지 않아야 한다.
- NFR-002. 대기열은 동시 사용자 5,000명을 목표로 해야 한다.
- NFR-003. 모든 API 응답시간은 p95 기준 1초 미만을 유지해야 한다.
- NFR-004. 모든 API는 통일된 처리 목표로 API별 100 TPS를 기준으로 설계해야 한다.
- NFR-005. 사용자에게 노출되는 오류 메시지는 내부 사유를 과도하게 드러내지 않고 `사용자 오류` 또는 `서버 오류` 수준으로 축약해야 한다.

### 4.2 동시성 제어
- NFR-006. 시스템은 동일 이벤트에 대한 동시 예매 상황에서 유효한 입장 토큰이 없는 사용자의 예매 생성과 예매 결제를 차단해야 한다.
- NFR-007. 시스템은 동일 좌석의 중복 판매를 방지해야 한다.
- NFR-008. 시스템은 대기열 승급, 입장 토큰 발급, 결제 완료, 결제 제한시간 만료가 동시에 발생해도 좌석과 예매 상태가 중복 확정되지 않도록 해야 한다.
- NFR-009. 시스템은 회원당 동시에 보유 가능한 결제 대기 예매를 전체 1개로 제한해야 한다.

### 4.3 데이터 정합성
- NFR-010. 시스템은 예매 상태, 결제 상태, 환불 상태를 명시된 상태값으로 일관되게 조회 가능해야 한다.
- NFR-011. 결제 승인 요청은 비동기 처리될 수 있으며, 사용자는 결제 상태 polling API로 최종 결과를 확인해야 한다.
- NFR-012. 결제 제한시간 초과 시 예매 자동 취소와 좌석 해제는 즉시 반영되어야 한다.
- NFR-013. 예매 생성 또는 결제 승인 커밋 시점에 이벤트 상태 변경이 감지되면 시스템은 이벤트 변경을 사용자에게 고지하고 해당 트랜잭션을 롤백해야 한다.
- NFR-014. 결제 승인 도중 예매가 만료되면 시스템은 결제를 취소해야 한다.

### 4.4 확장성
- NFR-015. Gateway, Eureka 기반 서비스 디스커버리, Redis, RabbitMQ, 다중 모듈 분리를 유지해 서비스 단위 확장이 가능해야 한다.
- NFR-016. 대기열 polling은 SWR, single-flight, server-side polling rate control을 전제로 확장해야 한다.
- NFR-017. 브로커, 결제, 이벤트, 좌석 서비스는 동일 API 처리 목표인 100 TPS와 p95 1초 미만 응답 목표를 기준으로 병목을 관찰하고 필요 시 수평 확장할 수 있어야 한다.

### 4.5 장애 격리 및 가용성
- NFR-018. 인증, 이벤트 조회, 대기열, 결제 기능은 분리된 서비스 장애가 전체 시스템 중단으로 즉시 확산되지 않도록 운영돼야 한다.
- NFR-019. SSE 대기열은 deprecated이므로 가용성 목표의 기준 경로는 polling 대기열로 둔다.
- NFR-020. 대기열 서비스 장애, 예매 만료, 결제 승인 처리 중 장애가 발생해도 사용자에게는 `서버 오류` 수준의 축약 메시지를 노출해야 한다.

### 4.6 보안
- NFR-021. 시스템은 인증이 필요한 엔드포인트에 대해 액세스 토큰 기반 인증과 역할 기반 권한 검사를 적용해야 한다.
- NFR-022. 시스템은 공개 엔드포인트와 보호 엔드포인트를 게이트웨이 화이트리스트로 구분해야 한다.
- NFR-023. 시스템은 매니저가 자신이 생성한 이벤트에 대해서만 수정, 삭제, 상태 변경, 구매 조회, 환불 처리를 수행하도록 권한을 제한해야 한다.
- NFR-024. 개인정보 암호화는 비밀번호에만 적용해야 한다.
- NFR-025. 관리자 백오피스 사용자 관리 기능은 관리자 역할을 가진 사용자에게만 허용해야 한다.
- NFR-026. 관리자 로그인 실패 횟수 제한은 3회로 한다.
- NFR-027. 관리자 로그인 실패가 3회 발생하면 관리자 계정을 잠가야 한다.
- NFR-028. 관리자 백오피스 인증/인가는 세션 방식으로 구현해야 하며, 관리자 세션은 30분 동안 비활성 상태이면 만료되어야 한다.

### 4.7 장애 대응 및 복구
- NFR-029. 사용자가 대기열에서 명시적으로 이탈하면 시스템은 다음 사용자 승급이 가능해야 한다.
- NFR-030. 결제 승인 비동기 처리 실패 또는 예매 제한시간 경과 시 사용자가 상태 조회를 통해 실패 또는 만료를 인지할 수 있어야 한다.
- NFR-031. 결제 제한시간이 경과한 예매는 자동 취소되고 좌석은 즉시 복구되어야 한다.
- NFR-032. 결제 또는 환불 처리가 서버 비정상 종료 등으로 실패하면 시스템은 DLQ를 사용해 실패 작업을 재처리할 수 있어야 한다.
- NFR-033. 이미지 업로드 또는 교체 저장이 실패하면 시스템은 3회 재시도해야 한다.
- NFR-034. 이미지 업로드 또는 교체 저장이 3회 재시도 후에도 실패하면 시스템은 대체 이미지를 노출해야 한다.
- NFR-035. 이미지 업로드 또는 교체 저장이 최종 실패하면 시스템은 사용자에게 축약 오류를 알려야 하며, 알림 기능은 추후 구현으로 둔다.

### 4.8 감사 추적 및 운영성
- NFR-036. 시스템은 Swagger/OpenAPI 문서와 Actuator health/info/prometheus 엔드포인트를 통해 운영 상태를 점검할 수 있어야 한다.
- NFR-037. 대기열 사용자 수, polling 요청률, 활성 입장 토큰 수, 결제 처리 상태, 예매 만료 수, 환불 처리 결과에 대한 운영 모니터링 지표가 필요하다.
- NFR-038. 관리자 백오피스에서 사용자 생성, 매니저 승급, 논리 삭제, 정지, 정지 해제를 수행한 이력은 감사 추적 가능해야 한다.
- NFR-039. 인증/인가가 필요한 API 호출은 요청 원문, 응답 원문, 에러 원문, 사용자 정보, 발생 시각을 감사 로그로 저장해야 한다.
- NFR-040. 감사 로그와 운영 로그는 1개월 동안 보존해야 한다.
- NFR-041. 기존 운영 로그가 있는 경우 유지하고, 부족한 운영 로그는 추가해야 한다.

## 5. 제약사항 및 가정
- 본 문서는 백엔드 API와 Thymeleaf 기반 관리자 백오피스 UI 요구사항을 다룬다. 일반 사용자 클라이언트 코드는 별도이며, 프론트엔드는 백엔드 API에 맞춰 조정한다.
- 클라이언트의 외부 진입점은 게이트웨이(`platform/gateway`)라고 가정한다.
- `/api/v1/**`, `/static/**`, polling 대기열 엔드포인트 외의 내부 전용 API는 본 문서 범위에서 제외한다.
- 알림 기능은 현재 구현하지 않으며, 이벤트 스토밍 범위에서도 제외한다.
- SSE 기반 대기열은 deprecated이며 공식 경로가 아니다.
- 3D/렌더링 기능은 사용하지 않는다.
- 클라이언트 테스트와 E2E 테스트는 이후 단계로 미룬다.

## 6. 기반 기술 결정
### 6.1 구현 스택
- 클라이언트 프레임워크: 백엔드 요구사항 범위 밖. 클라이언트 코드는 별도이며 프론트엔드가 백엔드에 맞춘다.
- 언어: Java 21
- 빌드 도구: Gradle 멀티 모듈
- 3D/렌더링 라이브러리: 사용하지 않음
- 테스트 도구: 백엔드는 JUnit Platform 및 Spring Boot 테스트 도구를 사용한다. 클라이언트 테스트/E2E는 이후 단계로 지연한다.
- 저장소/런타임 제약: Spring Boot 기반 마이크로서비스 구조, Gateway + Eureka, MySQL(primary/readonly), Redis, RabbitMQ 사용을 전제로 한다.

### 6.2 아키텍처/운영 결정
- 저장 방식:
  - MySQL: 서비스 데이터 저장
  - Redis: 대기열, 입장 토큰 검증, 일부 조회수/실시간 상태
  - 이벤트 이미지: 로컬 정적 파일 저장과 외부 정적 이미지 저장 옵션을 모두 구현한다. dev profile은 local 저장을 사용한다.
- 통합 방식:
  - 외부 클라이언트는 Gateway를 통해 접근한다.
  - 서비스 디스커버리는 Eureka를 사용한다.
  - 대기열 공식 경로는 HTTP polling이다.
  - 서비스 간 비동기 통합에는 RabbitMQ 사용을 전제로 한다.
- 외부 API 사용 여부:
  - Google, Kakao OAuth 사용
  - 결제 제공자는 Toss Payments만 사용
  - 기존 결제 코드를 우선 사용하고, 불명확한 부분은 인터페이스까지만 정의하며 신규 인프라 구현은 보류한다.
- 로깅/디버깅:
  - Swagger/OpenAPI 노출
  - Actuator health/info/prometheus 노출
  - 운영 로그와 디버그 로그 사용
  - 감사 로그 보존 기간은 1개월로 한다.
  - 관리자 백오피스 UI는 Thymeleaf로 구현한다.
