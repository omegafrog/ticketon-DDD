# 도메인 모델

## 1. 개요
- 입력 이벤트 스토밍 문서: docs/design/이벤트 스토밍.md
- 입력 유스케이스 문서: docs/design/유스케이스.md
- 입력 요구사항 문서: docs/design/요구사항.md
- 목적: 이벤트, 커맨드, 정책으로부터 엔티티, VO, 도메인 메서드, 도메인 서비스를 도출한다.
- 설계 준비 게이트:
  - 비즈니스 정책 확인 필요: 없음
  - 기반 기술 결정 확인 필요: 없음
  - 이벤트 스토밍 확인 필요: 없음

## 2. 도출 기준
- 엔티티 기준: 계정, 이벤트, 대기열 등록, 입장 토큰, 좌석, 예매, 구매/결제, 환불, 이미지, 관리자 세션, 감사 로그, 운영 로그, DLQ 작업처럼 식별자와 생명주기 상태가 필요하고 시간이 지나도 동일성을 추적해야 하는 개념을 엔티티로 둔다.
- VO 기준: 이메일, 비밀번호 해시, 역할, 계정 상태, 기간, 금액, 좌석 선택, 결제 제한시간, 결제 제공자 식별값, 이미지 파일 조건, 재시도 정책, 오류 노출 수준, 보존 기간처럼 값 자체의 유효성 검증과 불변 교체가 중요한 개념을 VO로 둔다.
- 메서드 기준: 자기 상태를 변경하거나 자기 상태만 검증하는 행위만 엔티티/VO 메서드로 둔다. 다른 BC, 저장소, 외부 결제 제공자, 파일 저장소, 메시지 브로커, 현재 시간, 감사/운영 로그 수집과 협력해야 하는 행위는 도메인 서비스 후보로 분리한다.
- 도메인 서비스 기준: 여러 모델의 상태를 함께 판단하거나 외부 협력 결과가 필요한 정책, 예를 들어 대기열 승급, 좌석 중복 판매 방지, 예매-결제-이벤트 변경 충돌, 결제 만료 보상, 환불 가능성, DLQ 재처리, 감사 로그 캡처, 이미지 저장 재시도는 도메인 서비스 후보로 둔다.
