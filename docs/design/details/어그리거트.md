# 어그리거트

## 1. 개요
- 입력 도메인 모델 문서: docs/design/details/도메인모델.md
- 입력 요구사항 문서: docs/design/요구사항.md
- 입력 유스케이스 문서: docs/design/유스케이스.md
- 입력 이벤트 스토밍 문서: docs/design/이벤트 스토밍.md
- 목적: 원자적으로 변경되어야 하는 엔티티/VO 경계와 aggregate root를 정의한다.
- 설계 준비 게이트:
  - 비즈니스 정책 확인 필요: 없음
  - 기반 기술 결정 확인 필요: 없음
  - 이벤트 스토밍 확인 필요: 없음

## 2. 어그리거트 설계 원칙
- 원자적 변경 경계: 한 명령에서 반드시 함께 검증/전이되어야 하는 상태만 같은 어그리거트에 둔다. 예매-좌석-입장 토큰-결제처럼 여러 어그리거트가 함께 변하는 흐름은 애플리케이션 서비스가 순서와 트랜잭션/보상을 조정한다.
- 루트 엔티티 접근: 외부 코드는 루트 엔티티 메서드만 호출한다. 포함 엔티티인 `UserProfile`, `SocialAccountLink`, `AdminLoginFailureRecord`, `Seat`, `SeatLayout` 변경은 각각의 루트가 중재한다.
- setter 금지: 상태 전이는 `create`, `change`, `expire`, `confirm`, `cancel`, `mark*`, `record*` 같은 의도를 가진 루트 메서드로만 수행한다.
- 외부 협력 금지: OAuth, Toss Payments, 파일 저장소, Redis/RabbitMQ, HTTP 요청/응답 원문 캡처, 현재 시각 생성, repository 조회는 어그리거트 내부에서 호출하지 않는다. 필요한 값은 애플리케이션 서비스/포트/도메인 서비스가 구해 루트 메서드 인자로 전달한다.
- 성능/운영 관찰: p95 1초 미만 및 API별 100 TPS 목표는 개별 업무 어그리거트의 책임이 아니라 `OperationalLogAggregate`와 이후 애플리케이션 서비스/운영 포트의 관찰 제약으로 둔다.
