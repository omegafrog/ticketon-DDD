# 애플리케이션 서비스

## 1. 개요
- 입력 유스케이스 문서: docs/design/유스케이스.md
- 입력 이벤트 스토밍 문서: docs/design/이벤트 스토밍.md
- 입력 어그리거트 문서: docs/design/details/어그리거트.md
- 목적: 유스케이스별 흐름을 오케스트레이션하는 애플리케이션 서비스를 정의한다.
- 설계 준비 게이트:
  - 비즈니스 정책 확인 필요: 없음
  - 기반 기술 결정 확인 필요: 없음
  - 이벤트 스토밍 확인 필요: 없음

## 2. 설계 원칙
- 유스케이스 단위 오케스트레이션: 애플리케이션 서비스는 인증 주체 식별, repository 조회, 도메인 서비스 판단 호출, aggregate root 메서드 호출, 저장, 포트 호출, 결과 조립 순서만 조정한다.
- Repository를 통한 Aggregate 조회/저장: 상태 변경은 항상 대상 aggregate를 repository로 조회한 뒤 root 메서드를 호출하고, 변경된 aggregate만 저장한다. 조회 전용 유스케이스는 query repository/projection을 사용해 p95 1초 미만과 API별 100 TPS 운영 관찰 대상이 되도록 한다.
- Aggregate Root 메서드 호출: 하위 엔티티/컬렉션을 직접 수정하지 않고 `UserAccountAggregate`, `EventAggregate`, `QueueEntryAggregate`, `AdmissionTokenAggregate`, `SeatInventoryAggregate`, `ReservationAggregate`, `PurchaseAggregate`, `RefundAggregate`, `DeadLetterTaskAggregate`, `AuditLogAggregate`, `OperationalLogAggregate`의 root 메서드만 호출한다.
- 외부 협력 객체/다른 BC 호출: OAuth, Toss Payments, 파일 저장소, 대기열/입장 토큰, 좌석, 이벤트, 감사/운영 로그, DLQ/메시징 협력은 포트로 호출한다. 다른 BC의 내부 모델에는 의존하지 않고 식별자, 스냅샷, 판단 결과 값만 주고받는다.
- 비즈니스 로직 노출 금지: 3회 관리자 로그인 실패 잠금, 리프레시 토큰 무효화, 이벤트 작성자 권한, 이미지 3회 재시도 후 대체 이미지, 예매/결제 이벤트 변경 충돌, 예매 만료 중 결제 취소, 부분 환불 금지 같은 판단은 aggregate/domain service가 수행하고 애플리케이션 서비스는 판단 결과에 따른 순서만 조정한다.
- 트랜잭션/저장 순서 고려: 단일 aggregate 변경은 한 트랜잭션으로 저장한다. 여러 aggregate/BC가 함께 변하는 예매, 결제, 환불, 이미지, 로그, DLQ 흐름은 커밋 전 충돌 검증, 저장 후 포트 호출, 실패 시 보상 명령/DLQ 등록 순서로 설계한다.
