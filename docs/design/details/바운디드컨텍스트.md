## 2. 경계 결정 기준
- 같은 도메인 용어가 다른 모델로 표현되는 경우: `사용자`는 인증/인가의 계정, 사용자 정보의 프로필, 관리자 백오피스의 운영 대상 사용자로 의미가 다르므로 분리한다. `좌석`은 이벤트의 레이아웃 입력값, 좌석 재고의 점유/판매 상태, 예매의 선택 좌석 스냅샷으로 다르게 표현한다. `이벤트`는 공개 조회, 매니저 운영, 이미지가 같은 판매/노출 규칙과 이벤트 버전을 공유하므로 하나의 Event BC로 합친다.
- 규칙 변경 시점이 다른 경우: 인증 토큰/계정 잠금, 사용자 프로필, 관리자 세션, 이벤트 판매 정책, 대기열 승급, 좌석 재고, 예매/결제/환불, 감사/운영 로그는 서로 다른 변경 주기와 장애 격리 요구를 갖는다.
- 함께 변경되어야 하는 규칙: reservation, purchase/payment, refund는 결제 제한시간, 금액 일치, 이벤트 변경 충돌, 예매 만료 중 결제 취소, DLQ 재처리 정책을 공유하므로 Booking & Commerce BC로 합친다. 공개 이벤트 조회, 이벤트 운영, 이벤트 이미지는 OPEN/DELETED 노출, 작성자 권한, 이미지 fallback, 이벤트 버전 변경을 공유하므로 Event BC로 합친다.
- 외부 협력 또는 통합 경계: OAuth/토큰 발급, Toss Payments, 파일 저장소, RabbitMQ/DLQ, 감사 로그 캡처, 운영 지표 수집은 포트로 분리한다. 다른 BC의 내부 모델은 직접 수정하지 않고 식별자, 스냅샷, 판단 결과, 명령/이벤트만 교환한다.
