## 3. 정본 경계

### 3.1 Bounded Context

| Bounded Context | 책임 | 보유 요소 | 근거 |
|---|---|---|---|
| `Access Control Context` | 인증 판정, 생성 권한 판정 | `AuthenticationDecision`, `AuthorizationDecision` | 모든 UC가 `Gateway 인증 계층` 선행을 요구하고, `UC-032`만 `ADMIN` 또는 `MANAGER` 권한 판정을 추가로 요구한다. |
| `Notification Management Context` | `Notification` 생성, 조회, 읽음 전이, 삭제 | `NotificationAggregate` | `UC-030`, `UC-031`, `UC-032`의 도메인 규칙이 모두 recipient-scoped persisted `Notification` lifecycle에 묶인다. |

### 3.2 Canonical Aggregate

| 구분 | 이름 | 정본 정의 | Provenance |
|---|---|---|---|
| Aggregate | `NotificationAggregate` | 하나의 persisted `Notification`을 root로 가지며 생성, 조회 후 읽음 전이, 삭제를 같은 언어와 invariant로 통합한다. | `UC-030`, `UC-031`, `UC-032` |
| Root Entity | `Notification` | `notificationId`, `recipientUserId`, `type`, `content`, `sentAt`, `readState`, `sourceKey`를 가진다. 삭제는 soft state가 아니라 aggregate 제거다. | `UC-030`, `UC-031`, `UC-032` |
| Value Object | `UserId` | recipient ownership과 recipient addressing을 표현하는 shared identifier다. | `UC-030`, `UC-031`, `UC-032` |
| Value Object | `NotificationContent` | `title`, `content`, `targetUrl`를 묶는 shared payload다. `content`는 required, `targetUrl`만 optional이다. | `UC-030`, `UC-032` |
| Value Object | `NotificationReadState` | `{UNREAD, READ}` 값 집합만 허용한다. 상세 조회 성공 전이는 `UNREAD -> READ` 단건 전이만 허용한다. | `UC-030`, `UC-032` |
| Value Object | `NotificationSelection` | `selected-set` 삭제 전용 값 타입이다. 중복 제거된 `notificationIds` 집합을 표현한다. | `UC-031` |
| Domain Service | `NotificationDomainService` | 유효한 생성 요청으로 unread `Notification` 1건을 구성한다. | `UC-032` |
| Domain Service | `NotificationDeletionPolicy` | foreign-owned 포함 거절, 이미 없는 owned 제외, 남은 owned만 삭제 규칙을 집합 단위로 적용한다. | `UC-031` |

## 4. 정본 동작 계약

### 4.1 Commands

| 분류 | Command | 의미 |
|---|---|---|
| 생성 | `CreateNotification` | 승인된 발신자가 지정 recipient에 unread `Notification` 1건을 만든다. |
| 조회 | `GetNotificationInbox` | recipient 범위 `Notification Inbox` 목록을 최신순 페이지로 반환한다. |
| 조회 | `GetUnreadNotificationInbox` | recipient 범위 `Unread Notification` 목록만 반환한다. |
| 조회 | `GetUnreadNotificationCount` | recipient 범위 `Unread Notification` 개수만 반환한다. |
| 조회/전이 | `GetNotificationDetail` | recipient 소유 단건 상세를 반환하고, 성공 시 해당 root 하나만 `READ`로 전이한다. |
| 삭제 | `DeleteNotification` | recipient 소유 단건만 삭제한다. |
| 삭제 | `DeleteSelectedNotifications` | `selected-set` 삭제를 수행한다. foreign-owned 포함 시 전체 거절, 이미 없는 owned은 제외한다. |
| 삭제 | `DeleteAllOwnedNotifications` | 요청 시점 recipient owned 전체 범위를 삭제한다. |

### 4.2 Events

| 분류 | Event | 의미 |
|---|---|---|
| 생성 | `NotificationCreated` | 유효한 생성 요청으로 unread `Notification` 1건이 저장되었다. |
| 조회 | `NotificationInboxReturned` | latest-first paginated inbox 조회 결과가 반환되었다. |
| 조회 | `UnreadNotificationsReturned` | unread 목록 조회 결과가 반환되었다. |
| 조회 | `UnreadNotificationCountReturned` | unread 개수 조회 결과가 반환되었다. |
| 조회 | `NotificationDetailReturned` | recipient 소유 단건 상세가 반환되었다. |
| 상태 전이 | `NotificationReadStateChanged` | 성공한 상세 조회 직후 동일 root만 `UNREAD -> READ`로 전이되었다. |
| 삭제 | `NotificationDeleted` | owned `Notification` root가 삭제되었다. |
| 거절 | `NotificationAccessRejected` | 인증 부족, 미존재, 비소유 조회는 aggregate 변경 없이 거절되었다. |
| 거절 | `NotificationDeletionRejected` | 비인증 또는 foreign-owned 포함 삭제 요청은 aggregate 변경 없이 거절되었다. |
| 거절 | `NotificationCreationRejected` | 권한 부족 또는 필수 입력 누락 생성 요청은 aggregate 저장 없이 거절되었다. |

### 4.3 States

| State | 의미 | 근거 |
|---|---|---|
| `UNREAD` | 상세 조회 성공 전 상태 | `UC-030`, `UC-032` |
| `READ` | recipient가 상세 조회에 성공한 뒤 상태 | `UC-030` |
| `Deleted` 상태 없음 | 삭제는 persistent state 전이가 아니라 aggregate 제거다. | `UC-031` |

### 4.4 Invariants

- 모든 조회와 삭제 결과는 인증된 요청자의 `Recipient User ID` 범위 안에서만 적용된다.
- `Notification Inbox`는 별도 저장 entity가 아니라 recipient-scoped query view다.
- inbox 목록은 항상 latest-first paginated 결과여야 한다.
- unread 목록과 unread 개수 조회는 `NotificationReadState`를 변경하지 않는다.
- 성공한 상세 조회는 반환된 동일 root 하나만 `UNREAD -> READ`로 전이할 수 있다.
- 유효한 생성 요청은 지정 recipient에 대해 unread `Notification` 정확히 1건만 저장한다.
- 생성에서 `Recipient User ID`, `Notification type`, `title`, `content`는 required이고 `targetUrl`만 optional이다.
- 단건 삭제와 `selected-set` 삭제는 foreign-owned `Notification`이 하나라도 포함되면 거절된다.
- `selected-set` 삭제는 이미 없는 owned `Notification`을 실패 사유로 만들지 않고 남은 existing owned만 삭제한다.
- `selected-set` 삭제에서 남은 existing owned가 0건이어도 요청은 정상 종료된다.
- `all-owned` 삭제는 요청 시점 recipient owned 전체 범위에만 적용된다.
- 생성 권한 검증은 `Access Control Context` 선행 책임이다. `Notification Management Context`는 승인된 생성 요청만 처리한다.
